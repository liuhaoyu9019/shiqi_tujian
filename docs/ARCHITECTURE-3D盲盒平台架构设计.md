## 三、核心模块边界

| 微服务 | 职责 | 核心接口 |
|--------|------|----------|
| **user-service** | 登录注册、JWT鉴权、个人信息、会员等级 | `/api/user/**` |

## 四、核心技术方案

### 4.1 公平随机抽盒算法

```
方案：基于 SecureRandom + 服务端概率引擎

1. 用户发起抽盒请求 → draw-service 生成交易流水
2. 服务端使用 SecureRandom + 复合种子执行加权随机
3. 按盲盒系列配置的概率表：遍历概率累积值，rand <= cursor 命中
4. 生成开盒结果写入数据库，MQ异步通知藏品服务
5. 前端3D开盒动画仅为表现层，结果已由服务端确定

概率校验机制：
- 每100次抽盒输出概率偏差报告
- 实际掉落率 vs 配置概率 偏差 > 1% 自动告警
- 支持运营配置动态调整概率（需审核留痕）
```

### 4.2 3D渲染性能方案

```
PC Web:
- Three.js + React Three Fiber
- glTF/GLB 格式模型，Draco 压缩（体积减少80%）
- LOD（Level of Detail）分级加载，按距离切换模型精度
- requestAnimationFrame 节流，非激活页30fps→10fps

移动H5:
- 设备检测，低端机自动降级：骨骼动画→静态图+CSS动画
- 模型纹理分辨率降级（2K→512）
- Web Worker 解压 Draco 模型，主线程不阻塞

小程序:
- WebGL Canvas 模式渲染
- 分包加载3D模型（主包<2MB，3D分包独立）
- 预加载关键模型到本地缓存
```

### 4.3 高并发开盒方案（秒杀级）

```
热门系列上线时瞬时高并发 → 类比秒杀场景：

1. 前端：按钮防抖 + 排队倒计时
2. 网关层：Sentinel 令牌桶限流（QPS阈值可动态调整）
3. 服务层：Redis 预扣库存 + Lua脚本原子操作
4. 异步：RocketMQ 削峰，开盒请求入队列排队处理
5. 兜底：超卖检查 + 自动退款 + 异常补偿
6. 防重：Redis分布式锁（userId维度，5秒过期）
```

### 4.4 数据流转核心链路

```
用户开盒:
  前端点击 → API Gateway(鉴权) → wallet-service(扣币)
  → draw-service(服务端定价+随机抽盒) → Redis(写入缓存)
  → RocketMQ(异步) → collect-service(入库藏品)
  → 前端返回结果 → 3D开盒动画播放

充值链路:
  前端下单 → wallet-service(创建订单) → 支付回调
  → wallet-service(Redis原子加币) → 前端更新余额
```

## 五、数据库架构

### 核心表结构

| 表名 | 说明 | 预估数据量 |
|------|------|-----------|
| t_user | 用户表 | 百万级 |
| t_wallet | 虚拟币钱包（乐观锁） | 百万级 |
| t_transaction | 交易流水 | 千万级 |
| t_box_series | 盲盒系列 | 百级 |
| t_item_def | 藏品定义 | 万级 |
| t_draw_record | 开盒记录 | 亿级（需分表） |
| t_user_item | 用户藏品 | 亿级（需分表） |
| t_recharge_order | 充值订单 | 千万级 |
| t_visit_log | 访问日志 | 亿级（需分表） |
| t_feed | 社交动态 | 千万级 |

### 索引策略

| 表 | 核心索引 | 原因 |
|----|----------|------|
| t_user | uk_phone, uk_openid | 登录查询唯一约束 |
| t_transaction | idx_user_created | 用户流水列表覆盖索引 |
| t_draw_record | idx_user_created, idx_series | 用户历史+系列统计 |
| t_user_item | uk_user_item | 查重+持有数量更新 |
| t_visit_log | idx_visit_date | 日期维度统计查询 |

### 缓存架构

```
Redis 数据结构设计:

├── string:  user:info:{uid}              → 用户信息缓存 (TTL 1h)
├── string:  wallet:balance:{uid}         → 钱包余额 (实时更新)
├── string:  series:config:{sid}          → 系列配置 (TTL 30min)
├── hash:    series:prob:{sid}            → 概率配置 (field=item_id)
├── zset:    rank:collect:{sid}           → 系列收集排行
├── zset:    rank:rare:{sid}              → 稀有藏品排行
├── set:     user:items:{uid}             → 用户已拥有藏品ID集合
├── list:    draw:queue:{sid}             → 高并发开盒排队队列
└── string:  lock:draw:{uid}              → 开盒分布式锁 (防重复提交)
```

## 六、安全架构

| 层级 | 措施 |
|------|------|
| 传输层 | HTTPS + CSP头 + CORS白名单 |
| 认证层 | JWT无状态Token + 恒定时间比较防时序攻击 |
| 应用层 | 参数校验(@Valid) + 服务端定价(防篡改) + XSS过滤 |
| 数据层 | AES加密敏感字段 + 脱敏展示 + 环境变量密码 |
| 运维层 | SecurityFilterChain + 日志脱敏 + DEBUG关闭 |

## 七、研发规范

- **命名规范**: Java驼峰、数据库snake_case、前端PascalCase组件
- **接口规范**: RESTful，统一响应体 `{code, message, data, timestamp}`
- **日志规范**: Log4j2 + TraceId全链路追踪
- **异常规范**: 全局异常拦截器 + 业务异常枚举 + 统一错误码
- **Git规范**: feat/fix/refactor 分支前缀，commit message中文

## 九、开发环境简化方案

当前开发阶段采用 **内嵌dev-server**（Vite插件）替代完整后端：
- 数据持久化：JSON文件
- 覆盖接口：17个API端点全覆盖
- 生产切换：修改 `vite.config.ts` 移除插件 + 配置Nginx代理 → Spring Boot
